# AlgoWiki 运行命令与数据库说明

## 1. 固定数据库路径（避免“数据丢失”）
请始终使用同一个数据库文件：

```powershell
$env:DB_ENGINE='sqlite'
$env:SQLITE_NAME='d:\AlgoWiki_Project\storage\db_live.sqlite3'
```

如果要启用站内 AI 助手，建议同时补充：

```powershell
$env:AI_ASSISTANT_ENCRYPTION_KEY='<一串随机的 urlsafe base64 密钥>'
$env:THROTTLE_ASSISTANT_ANON='15/hour'
$env:THROTTLE_ASSISTANT_USER='60/hour'
```

> 只要路径不变、文件不删，文章/公告/问答/评论/Issue/赛事时刻表等数据都会保留。

---

## 2. 一键初始化（首次或需要重建时）
> 会执行迁移并导入初始内容；`-Rebuild` 会清空旧库后重建。

```powershell
cd d:\AlgoWiki_Project
powershell -ExecutionPolicy Bypass -File .\scripts\init_database.ps1 `
  -DbPath "d:\AlgoWiki_Project\storage\db_live.sqlite3" `
  -SuperadminPassword "AlgoWiki@2026" `
  -Rebuild
```

### 管理员账号（执行上面命令后）
- 用户名：`superadmin`
- 密码：`AlgoWiki@2026`

---

## 3. 启动后端（后台非阻塞）
```powershell
cd d:\AlgoWiki_Project

  -DbPath "d:\AlgoWiki_Project\storage\db_live.sqlite3" `
```

## 7. 数据是否“全部走数据库”
当前后端模型中，这些功能数据都写入数据库表，不是前端临时状态：

- 知识库：`Category`、`Article`
- 评论：`ArticleComment`
- 修订：`RevisionProposal`
  - 额外保存 `base_title`、`base_summary`、`base_content_md`、`base_updated_at`
  - 用于多人同时修改同一篇 Wiki 时执行三方合并，尽量自动合并非重叠修改
- Issue/Request：`IssueTicket`
- 问答：`Question`、`Answer`
- 公告：`Announcement`、`AnnouncementRead`
- trick 技巧：`TrickEntry`
- 友链：`FriendlyLink`
- 赛事公告：`CompetitionNotice`
- 赛事时刻表：`CompetitionScheduleEntry`
- Team 展示：`TeamMember`
- AI 助手配置：`AssistantProviderConfig`
- AI 助手调用日志：`AssistantInteractionLog`

### 不丢失的前提
1. 永远使用同一个 `SQLITE_NAME` 路径。  
2. 不执行 `-Rebuild`（除非你要重置数据）。  
3. 不手动删除 `db_live.sqlite3` 文件。  
4. 建议定期备份（见下）。

### 备份与恢复
```powershell
# 备份
Copy-Item d:\AlgoWiki_Project\storage\db_live.sqlite3 d:\AlgoWiki_Project\storage\backup_$(Get-Date -Format yyyyMMdd_HHmmss).sqlite3

# 恢复（示例）
Copy-Item d:\AlgoWiki_Project\storage\backup_20260304_220000.sqlite3 d:\AlgoWiki_Project\storage\db_live.sqlite3 -Force
```

## 8. 生产环境运维手册（algowiki.cn）

### 当前正式入口
- 主站唯一入口：`https://www.algowiki.cn`
- 裸域跳转：`https://algowiki.cn -> 301 -> https://www.algowiki.cn/`
- CDN 只挂在：`www.algowiki.cn`
- 裸域 `algowiki.cn` 继续直连 ECS，用于稳定做 `301`

### 当前正式发布状态
- 服务器路径：`/srv/algowiki`
- 当前镜像：`ghcr.io/nullresot/algowiki-web:sha-bed3c97`
- 当前 release：`ghcr-bed3c97-links-calendar`
- 健康检查：`https://www.algowiki.cn/api/health/`

### 域名 / CDN / HTTPS 最终结构
```text
用户 -> https://www.algowiki.cn -> 阿里云 CDN -> https 回源到 algowiki.cn:443 -> Nginx -> 127.0.0.1:8001
用户 -> https://algowiki.cn -> ECS Nginx 301 -> https://www.algowiki.cn/
```

### Nginx 目标行为
- `80` 端口：`algowiki.cn` 和 `www.algowiki.cn` 一律重定向到 `https://www.algowiki.cn`
- `443` 端口：
  - `algowiki.cn`：只做 `301`
  - `www.algowiki.cn`：反向代理到 `127.0.0.1:8001`
- 证书路径：
  - `/etc/letsencrypt/live/algowiki.cn/fullchain.pem`
  - `/etc/letsencrypt/live/algowiki.cn/privkey.pem`

### Django 生产安全配置
- `SECURE_SSL_REDIRECT=1`
- `SESSION_COOKIE_SECURE=1`
- `CSRF_COOKIE_SECURE=1`
- `DJANGO_SECURE_PROXY_SSL_HEADER=HTTP_X_FORWARDED_PROTO,https`
- `DJANGO_USE_X_FORWARDED_HOST=1`
- `DJANGO_USE_X_FORWARDED_PORT=1`
- `DJANGO_ALLOWED_HOSTS=algowiki.cn,www.algowiki.cn,127.0.0.1,localhost`
- `DJANGO_CSRF_TRUSTED_ORIGINS=https://algowiki.cn,https://www.algowiki.cn`

### CDN 当前应保持的配置
- 加速域名：`www.algowiki.cn`
- 源站地址：`algowiki.cn`
- 回源 Host：`www.algowiki.cn`
- 回源协议：`HTTPS`
- 回源端口：`443`
- CDN 证书：使用包含 `algowiki.cn`、`www.algowiki.cn` 的证书

### CDN 缓存规则
- `/assets/`：缓存 `1 个月`
- `/static/`：缓存 `1 个月`
- `/wiki-assets/`：缓存 `1 个月`
- `/api/`：`0 秒`，不缓存
- `/admin/`：`0 秒`，不缓存
- `/manage/`：`0 秒`，不缓存
- `/review/`：`0 秒`，不缓存
- `/profile/`：`0 秒`，不缓存
- 首页 HTML：不要做长缓存

### 费用止损与预警
- CDN 流量封顶：`每小时 10 GB`，`解封时间 1 天`
- CDN HTTPS 请求数封顶：`每小时 200 万次`，`解封时间 1 天`
- 带宽封顶：暂不启用
- CDN 高额消费预警：`100 元`
- 异常检测：已开启
- 预算提醒：当前按 `50% / 80% / 100%` 三档提醒即可

### 线上验收命令
在服务器执行：

```bash
curl -I https://algowiki.cn
curl -I https://www.algowiki.cn
curl -fsS https://www.algowiki.cn/api/health/
```

预期结果：
- 第一条返回 `301`，并且 `location: https://www.algowiki.cn/`
- 第二条返回 `200`，并且响应头中出现 `server: Tengine`
- 第三条返回 JSON，并且 `"status":"ok"`

### 服务器综合巡检
在服务器执行：

```bash
cd /srv/algowiki
chmod +x deploy/server-verify.sh

```

巡检应重点确认：
- 容器健康状态正常
- `public site` 返回 `200`
- `apex redirect` 返回 `301`
- `/api/health/` 返回 `status: ok`

### 从镜像仓库更新正式环境
当 GitHub Actions 已推送新镜像到 GHCR 后，在服务器执行：

```bash
cd /srv/algowiki
chmod +x deploy/*.sh
./deploy/server-update-from-registry.sh --image ghcr.io/nullresot/algowiki-web:latest --release ghcr-latest
sleep 25
docker ps --format 'table {{.Names}}\t{{.Image}}\t{{.Status}}'
curl -fsS https://www.algowiki.cn/api/health/
```

如果要指定某个固定提交镜像，可改为：

```bash
./deploy/server-update-from-registry.sh --image ghcr.io/nullresot/algowiki-web:sha-<commit> --release ghcr-<commit>
```

如果这次上线要严格跟随 GitHub 仓库里已经合并到 `main` 的 PR 代码，推荐直接执行：

```bash
cd /srv/algowiki
./deploy/server-update-from-registry.sh --sync-github-branch --dry-run
./deploy/server-update-from-registry.sh --sync-github-branch
```

说明：

- `--sync-github-branch` 会先查询 `NullResot/AlgoWiki` 当前 `main` 的最新提交，再部署对应的 `sha-<commit>` 镜像
- 这样即使 `deploy/.env.production` 里之前固定过旧的 `APP_IMAGE`，下次上线也会跟随 GitHub 已合并的 PR 结果
- 如果 `docker pull` 报找不到镜像，通常说明 GitHub Actions 还没把这个提交的镜像推到 GHCR，等 `Publish GHCR Image` 变绿后重试即可
- 如果希望以后继续执行原来的 `./deploy/server-update-from-registry.sh` 也默认走这条逻辑，可在 `deploy/.env.production` 中补上 `DEPLOY_SYNC_GITHUB_BRANCH=1`、`GITHUB_REPOSITORY=NullResot/AlgoWiki`、`GITHUB_BRANCH=main`

### 从本地镜像包快速更新正式环境
如果这一次更新不想等服务器慢慢从 `ghcr.io` 拉镜像，可以直接在本地 Windows 上执行：

```powershell
cd D:\AlgoWiki_Project
powershell -ExecutionPolicy Bypass -File .\scripts\quick-release-to-server.ps1 `
  -ServerHost 139.224.212.247 `
  -ServerUser root
```

这条链路会自动完成：

- 本地构建 `algowiki-web:quick-<git-sha>`
- 导出镜像包到 `storage/deploy/`
- 上传镜像包到服务器
- 上传 `docker-compose.server.yml`、`deploy/server-compose-up.sh`、`deploy/server-update-from-archive.sh`
- 服务器执行 `deploy/server-update-from-archive.sh`

如果要自定义 SSH 私钥路径、端口或保留服务器上的镜像包，可用：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\quick-release-to-server.ps1 `
  -ServerHost 139.224.212.247 `
  -ServerUser root `
  -ServerPort 22 `
  -SshKeyPath "$HOME\\.ssh\\id_ed25519" `
  -KeepRemoteArchive
```

服务器侧也可以单独执行：

```bash
cd /srv/algowiki
chmod +x deploy/server-compose-up.sh deploy/server-update-from-archive.sh
./deploy/server-update-from-archive.sh --env-file deploy/.env.production --archive /root/algowiki-web-quick-<tag>.tar --image algowiki-web:quick-<tag> --release archive-quick-<tag> --cleanup-archive
```

```cron

*/30 * * * * /usr/bin/flock -n /tmp/algowiki-calendar-sync.lock /bin/bash -lc 'cd /srv/algowiki && docker-compose -f docker-compose.server.yml --env-file deploy/.env.production exec -T web python manage.py sync_competition_calendar' >> /var/log/algowiki-calendar-sync.log 2>&1
```

```bash

```

### 常见异常判断
- `curl -I https://www.algowiki.cn` 返回 `server: Tengine`
  - 说明 `www` 已经过 CDN
- `curl -I https://algowiki.cn` 返回 `301`
  - 说明裸域跳转正常
- `x-cache: MISS`
  - 首次访问或访问 HTML 时常见，未必异常
- `api/health` 返回 `status: ok`
  - 说明后端、数据库、静态资源和生产安全配置基本正常

### 不建议改动的项
- 不要把 `algowiki.cn` 也接到 CDN 上
- 不要把首页 HTML 做长缓存
- 不要把 `/api/`、`/manage/`、`/review/`、`/profile/` 做缓存
- 不要随意上调 CDN 封顶阈值，尤其在没有稳定访问量统计前

## 9. AI 助手补充说明

### 当前实现方式
- 前端右下角有一个可拖动的悬浮按钮，点击后可打开聊天面板。
- 助手只会根据站内公开内容回答，不做站外搜索。
- 回答会尽量附上站内来源链接，便于用户继续查看原文。

### 后台配置入口
- 管理后台入口：`/manage/assistant`
- 只有超级管理员可以新增配置、切换默认模型、测试连接、更新 API Key。
- 普通管理员可以查看配置状态和调用统计，但不能修改配置。

### API Key 安全规则
- API Key 只会以 `****************` 的形式显示。
- 后端数据库里保存的是加密后的密文，不保存明文。
- 如果你更换了 `DJANGO_SECRET_KEY`，建议同时重新配置 `AI_ASSISTANT_ENCRYPTION_KEY` 并重新录入 API Key。

### 推荐上线步骤
1. 先在服务器环境变量或 `backend/.env` 中设置 `AI_ASSISTANT_ENCRYPTION_KEY`。
2. 执行 `python manage.py migrate`，确保 AI 助手相关表已创建。
3. 使用超级管理员登录后台，在 `AI 助手` 页面填写 `Base URL`、`Model` 和 API Key。
4. 先点一次“测试连接”，成功后再打开公开悬浮助手。

### 修订并发合并上线补充
- 并发修订自动合并依赖迁移：`0031_revisionproposal_base_content_md_and_more`
- 如果是手动发布后端代码，发布后至少执行一次：
  - `python manage.py migrate`
  - `python manage.py showmigrations wiki`
- 确认 `0031_revisionproposal_base_content_md_and_more` 前面是 `[X]`，再去验证多人同时修改同一篇文章的场景

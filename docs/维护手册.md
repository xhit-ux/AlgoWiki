# AlgoWiki 维护手册

## 1. 适用对象

本手册面向：

- 服务器运维人员
- 项目负责人
- 后端维护者
- 负责值守和应急处理的同学

## 2. 维护目标

本项目维护的核心目标有三项：

1. 用户信息安全
2. 服务持续可用
3. 问题发生后可快速定位与恢复

## 3. 推荐生产环境

### 3.1 服务器建议

- CPU：2 核起步，推荐 2 核 4G
- 系统：Ubuntu 22.04 LTS / 24.04 LTS
- 磁盘：60G 以上 SSD
- 数据库：MySQL 8.x
- 运行方式：Docker Compose 或 Gunicorn + Nginx
- 如果只有 2G 左右内存，建议额外开启 2G 到 4G swap，并避免在服务器本机构建镜像

### 3.2 目录规划

建议部署到：

```text
/srv/algowiki
  ├─ backend/
  ├─ frontend/
  ├─ storage/
  │   └─ logs/
  └─ venv/
```

## 4. 关键文件

- 后端配置：`backend/config/settings.py`
- 环境变量示例：`backend/.env.example`
- 生产环境示例：`deploy/env.production.example`
- 私有验证环境示例：`deploy/env.private-validation.example`
- Docker 启动：`docker-compose.server.yml`
- 容器入口：`deploy/docker-entrypoint.sh`
- 服务器基线脚本：`deploy/server-bootstrap.sh`
- 服务器启动脚本：`deploy/server-compose-up.sh`
- 服务器验收脚本：`deploy/server-verify.sh`
- 本地镜像导出脚本：`scripts/build_server_image.ps1`
- 本地源码打包脚本：`scripts/package_server_source.ps1`
- systemd 模板：`deploy/gunicorn.service.example`
- Nginx 模板：`deploy/nginx.algowiki.conf`
- 日志轮转示例：`deploy/logrotate.algowiki.example`

## 5. 环境变量基线

生产环境至少确认以下变量：

### 5.1 基础安全

- `DJANGO_DEBUG=0`
- `DJANGO_SECRET_KEY=<长随机字符串>`
- `DJANGO_ALLOWED_HOSTS=<域名或公网IP>`
- `DJANGO_CSRF_TRUSTED_ORIGINS=<https域名>`

### 5.2 HTTPS / 反向代理

- `DJANGO_USE_X_FORWARDED_HOST=1`
- `DJANGO_USE_X_FORWARDED_PORT=1`
- `DJANGO_SECURE_PROXY_SSL_HEADER=HTTP_X_FORWARDED_PROTO,https`
- `SECURE_SSL_REDIRECT=1`
- `SECURE_HSTS_SECONDS=31536000`
- `SESSION_COOKIE_SECURE=1`
- `CSRF_COOKIE_SECURE=1`

### 5.3 数据库

- `DB_ENGINE=mysql`
- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `DB_CHARSET=utf8mb4`

### 5.4 日志与巡检

- `DJANGO_LOG_LEVEL=INFO`
- `DJANGO_LOG_DIR=/srv/algowiki/storage/logs`
- `DJANGO_REQUEST_LOG_ENABLED=1`
- `DJANGO_SLOW_REQUEST_MS=1500`
- `HEALTH_MIN_DISK_FREE_MB=1024`

## 6. 部署方式

### 6.1 Docker Compose 部署

#### 步骤

1. 在本地通过 VS Code Remote-SSH 打开 `/srv/algowiki`
2. 在服务器执行 `deploy/server-bootstrap.sh`
3. 在本地构建并导出镜像，再打包源码
4. 上传源码包和镜像包到服务器
5. RDS 未准备好前，用 `deploy/env.private-validation.example` 做私有验证
6. RDS 就绪后，用 `deploy/env.production.example` 切换正式配置
7. 备案通过后，再接入域名、Nginx 和 HTTPS

#### 低内存服务器建议

```bash
chmod +x deploy/server-bootstrap.sh
SWAP_MB=4096 CONFIGURE_UFW=1 ./deploy/server-bootstrap.sh
```

不建议在 2G 左右内存的服务器上执行：

```bash
docker compose up -d --build
```

更稳妥的方式是：

1. 本地执行 `scripts/build_server_image.ps1`
2. 本地执行 `scripts/package_server_source.ps1`
3. 服务器执行 `deploy/server-compose-up.sh --image-archive ...`

#### 私有验证示例

```bash
cd /srv/algowiki
cp deploy/env.private-validation.example deploy/.env.private-validation
chmod +x deploy/server-compose-up.sh
./deploy/server-compose-up.sh --env-file deploy/.env.private-validation --profile local-db --image-archive /root/algowiki-web-20260312.tar
```

#### 正式部署示例

```bash
cd /srv/algowiki
cp deploy/env.production.example deploy/.env.production
./deploy/server-compose-up.sh --env-file deploy/.env.production --image-archive /root/algowiki-web-20260312.tar
```

#### 说明

- `docker-compose.server.yml` 中的 `db` 服务只有在 `--profile local-db` 时才会启动
- 正式环境默认不再使用本机 MySQL 容器，而是连接 RDS 私网地址
- `APP_BIND` 默认是 `127.0.0.1`，建议始终通过 Nginx 暴露公网流量
- 若服务器位于中国内地且无法直连 OpenAI，可固定采用“本地 Codex + Remote-SSH”的协作方式
- 详细执行顺序见《远程部署指南》

### 6.2 Gunicorn + Nginx 部署

适合传统 Linux 环境。

建议步骤：

1. 安装 Python 依赖
2. 安装 Node 依赖并构建前端
3. 执行 `python manage.py check`
4. 执行 `python manage.py migrate`
5. 执行 `python manage.py collectstatic`
6. 启用 Gunicorn systemd 服务
7. 启用 Nginx 配置

## 7. 日志体系

### 7.1 日志文件

默认日志目录：

`storage/logs/`

主要日志：

- `algowiki.log`：应用日志、请求日志、一般异常
- `security.log`：登录失败、锁定、封禁、改密等安全事件

### 7.2 请求追踪

系统会为每个请求生成 `X-Request-ID`。

作用：

- 让前端报错、Nginx 日志、Django 日志可以串起来
- 用户反馈问题时，可以要求其提供该请求 ID

### 7.3 慢请求

当请求耗时超过 `DJANGO_SLOW_REQUEST_MS` 时，会以更高等级输出到日志。

建议：

- 发现慢请求后优先查 SQL、导出操作和后台批量任务

## 8. 健康检查与巡检

### 8.1 健康检查接口

`GET /api/health/`

关注字段：

- `database.ok`
- `media.ok`
- `storage.ok`
- `frontend.ok`
- `status`
- `request_id`

### 8.2 日常巡检项

建议每天或每次发布后至少检查：

1. `GET /api/health/` 是否为 `200`
2. 首页是否可打开
3. 登录、登出、修改密码是否正常
4. 一次文章访问、一次问题访问是否正常
5. `storage/logs/` 是否持续写入
6. 服务器磁盘剩余空间是否充足
7. MySQL 是否可连接

### 8.3 发布后检查

发布完成后建议执行：

```bash
python backend/manage.py check
python backend/manage.py check --deploy
python backend/manage.py showmigrations
```

## 9. 数据库维护

### 9.1 备份建议

生产环境至少保留：

- 每日全量备份
- 最近 7 天备份
- 每次发布前额外备份

推荐命令示例：

```bash
mysqldump -h 127.0.0.1 -P 3306 -u algowiki -p --single-transaction --default-character-set=utf8mb4 algowiki > algowiki_$(date +%F_%H%M%S).sql
```

### 9.2 恢复建议

恢复前必须：

1. 停止写入流量
2. 备份当前故障库
3. 明确恢复时间点

恢复示例：

```bash
mysql -h 127.0.0.1 -P 3306 -u algowiki -p algowiki < backup.sql
```

### 9.3 迁移管理

上线前必须确认：

- 迁移文件已纳入版本控制
- 生产库备份已完成
- 迁移在测试环境验证通过

禁止在生产环境直接手改表结构但不补迁移文件。

## 10. 媒体文件维护

上传文件位于：

- 开发环境：`backend/media/`
- 生产环境：由 `MEDIA_ROOT` 指定

维护要求：

- 媒体目录必须可写
- 定期检查容量
- 定期扫描异常大文件
- 重要环境建议纳入备份

## 11. 常见故障处理

### 11.1 `/api/health/` 返回 503

排查顺序：

1. 看响应里是 `database`、`media`、`storage` 还是 `frontend` 异常
2. 查看 `storage/logs/algowiki.log`
3. 查看 MySQL 连接状态
4. 检查磁盘空间
5. 检查 `frontend/dist/index.html` 是否存在

### 11.2 无法登录

排查：

1. 查看 `security.log`
2. 检查是否触发登录锁定
3. 检查账号是否被封禁/软删除
4. 检查时间同步是否异常导致 Token TTL 判断偏差

### 11.3 图片上传失败

排查：

1. 检查 `MEDIA_ROOT` 是否存在且可写
2. 检查磁盘空间
3. 检查上传文件大小和类型
4. 检查 Nginx 或代理的上传大小限制

### 11.4 发布后首页 404 或空白

排查：

1. `DJANGO_SERVE_FRONTEND=1` 是否开启
2. `FRONTEND_DIST_DIR` 是否指向正确目录
3. 是否执行过前端构建
4. 是否执行过 `collectstatic`

## 12. 安全维护要求

### 12.1 必做项

- 使用强随机 `SECRET_KEY`
- 所有生产站点启用 HTTPS
- 不在日志、文档或截图中泄露数据库口令
- 不将 `backend/.env` 提交到仓库
- 定期更换管理员密码
- 限制 MySQL 仅允许可信主机访问

### 12.2 建议项

- 数据库启用 TLS
- 服务器启用自动安全更新
- 使用非 root 用户运行应用
- 启用最小权限原则
- 给管理员账号单独建立使用规范

## 13. 升级与回滚

### 13.1 升级前

1. 备份数据库
2. 备份 `.env`
3. 记录当前版本号/提交号
4. 在测试环境验证迁移和关键流程

### 13.2 升级后

1. 检查 `/api/health/`
2. 检查登录、Wiki、问答、后台
3. 检查日志是否有新异常

### 13.3 回滚原则

代码回滚前先确认：

- 是否已经执行了破坏性迁移
- 是否需要同时回滚数据库
- 是否保留了升级前备份

## 14. 值守建议

如果项目长期面向公网运行，建议建立最小值守制度：

- 每天一次健康检查
- 每周一次备份恢复抽查
- 每次发布后检查日志
- 每月检查管理员账号和权限
- 每月检查磁盘、数据库容量、媒体增长情况

## 15. 建议的后续增强

1. 接入 Redis 做统一限流与缓存
2. 接入 Prometheus/Grafana 或云监控
3. 增加自动备份脚本和恢复演练脚本
4. 增加告警渠道
5. 拆分超大视图与超大管理页面，降低单点修改风险
